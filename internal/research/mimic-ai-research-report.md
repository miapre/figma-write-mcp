# Teaching Mimic to See a Design System

**Mimic AI already solves the hardest economic problem in HTML-to-Figma conversion — the Figma read-quota bottleneck — through a local learning cache, but its transparency, memory architecture, and component-matching pipeline lag behind what's now state-of-the-art in AI copilots and agent memory frameworks.** This matters because Mimic's core thesis ("an MCP that builds in Figma using your design system and gets faster every time") is *the* underserved direction in the Figma-AI ecosystem: every well-funded competitor — Builder.io Visual Copilot, Anima, Figma Make, Locofy — runs the other way (Figma → code). Mimic has a defensible niche if it compounds learnings transparently and matches components with higher fidelity than brittle heuristics. The research below concludes that Mimic should adopt (a) a Graphiti-style non-destructive, bi-temporal learning store keyed on pattern signatures, (b) a Code Connect–inspired catalog for component matching with hybrid retrieval, and (c) a ChatGPT+Cursor-blended UI pattern for surfacing what it has learned.

## The HTML-to-Figma landscape is thin where Mimic plays

Mimic AI is an independent, open-source MCP server — not a Figma product — built for Claude Code and documented at glama.ai/mcp/servers/miapre/mimic-ai. Its stack is `Claude Code → MCP server (mcp.js) → bridge.js → WebSocket → Figma plugin → canvas`. Architecturally it is built around a **reads-vs-writes asymmetry**: Figma's REST/MCP reads are quota-limited (200/day Professional, 600/day Enterprise, shared with all Claude activity), while plugin writes are unlimited. Mimic's Instant mode uses **3–5 reads on a cold build, ~1 on warm, and zero on a fully-learned build**, caching everything in a local `ds-knowledge.json` file that records HTML-pattern-to-component and variable-ID-to-token mappings. Every run reports **patterns saved, patterns promoted to VERIFIED, reads used, and DS gaps detected** — the last being a useful DS-audit byproduct. When the remaining budget would risk an incomplete build, Mimic stops rather than burn reads on retries. These are the most differentiated features in the space.

Every other notable tool moves in the opposite direction or ignores design systems. **Builder.io Visual Copilot** imports Figma into code using a Mitosis IR and can map against a catalog, but its "Figma-to-code" framing dominates. **Anima** and **Locofy.ai** are squarely Figma-to-code. **html.to.design (Divriots)** and **bernaferrari/FigmaToCode** produce raw frames without design-system awareness. **Figma Make** (GA after Config 2025) can use library styling context but as of mid-2025 testing was still ignoring corporate typefaces and component structures; Figma confirmed reading components was slated for fall 2025. **Figma's official MCP server** (mcp.figma.com/mcp) supports read/write with Dev-seat rate limits mirroring Tier-1 REST, and now generates design-system **rules files** describing tokens, components, style hierarchies, and naming conventions — a direct echo of what Mimic does in `ds-knowledge.json`. Of the many Figma MCP servers (Framelink/GLips, figma-mcp-go, southleft/figma-console-mcp for DS extraction and WCAG auditing, sunnysideFigma, claude-talk-to-figma-mcp), **none implement Mimic's pattern of persistent cross-run learning**. The closest adjacent projects are **open-pencil** (open-source Figma alternative with 90+ AI tools and MCP) and **Anil4Guney/figma-to-ai-html** (which reverses Mimic's idea — Figma to HTML that respects existing CSS variables). The competitive map has a clear empty quadrant: **code/HTML→Figma with design-system preservation and compounding learning**. That is Mimic's moat.

## Figma's APIs give Mimic more leverage than it currently uses

The Figma plugin API exposes a surprisingly rich surface for design-system-aware automation. Variables come in four types (`COLOR`, `FLOAT`, `STRING`, `BOOLEAN`) with **modes, collections, 20+ scopes** (e.g., `ALL_FILLS`, `CORNER_RADIUS`, `FONT_WEIGHT`), and `codeSyntax` hints that mirror Code Connect. The key binding primitives are `setBoundVariable(field, variableId)` on nodes and `figma.variables.setBoundVariableForPaint(paint, 'color', variable)` for fills/strokes; library variables are discovered via `figma.teamLibrary.getAvailableLibraryVariableCollectionsAsync()` → `getVariablesInLibraryCollectionAsync(collectionKey)` → `importVariableByKeyAsync(variableKey)`. Components are asymmetrical: there is **no team-library component enumeration API** parallel to variables, so Mimic (and everyone else) must enumerate via REST `GET /v1/files/:key/components` and `/component_sets`, cache the keys, then hydrate with `importComponentByKeyAsync(key)`. Component properties split into `BOOLEAN | TEXT | INSTANCE_SWAP | VARIANT | SLOT`, with non-variant names carrying `#id` suffixes (e.g., `ButtonText#0:1`). Instances accept `setProperties({...})` and `swapComponent(main)` without detaching overrides — the core primitives needed for high-fidelity HTML mapping.

**Code Connect** (figma.com/code-connect-docs/) is the underused multiplier. It links Figma components to real source components (`Button.figma.tsx` mapping `figma.string('Label')`, `figma.boolean('Disabled')`, `figma.enum('Size', {...})` to JSX props) and is what the Figma MCP server injects into LLM context so agents import `src/components/ui/Button.tsx` instead of inlining markup. **Mimic should invert the flow and consume Code Connect mappings as its canonical HTML→component dictionary**, because any design system with a Code Connect setup has already done the hardest work: declaring prop shapes and source-of-truth names. Atomic-design hierarchies (atoms → molecules → organisms → templates) plus the primitive/semantic/component token triad (Material's `md.ref → md.sys → md.comp`; Polaris's `space-100` primitives + `color-bg-surface` semantics; Primer's `scale.*` + `fg.default`) give Mimic a principled vocabulary to generalize patterns.

For the actual matching problem, five techniques exist in practice, and the pragmatic pipeline combines them: (1) **heuristic tag+class mapping** is fast but brittle; (2) **structural signatures** (DOM subtree shape, child count, flex direction, dimensions) give feature vectors; (3) **text-embedding similarity** over `{component_name + description + property names + variant options + docs}` supports top-k retrieval; (4) **LLM-in-the-loop** with a compact JSON catalog in context then emits `importComponentByKeyAsync(key) + setProperties({...})` calls; and (5) **multimodal matching** via `exportAsync({format:'PNG'})` + CLIP/GPT-4o vision catches visual similarity when names diverge. Builder.io's Visual Copilot and the Figma MCP server both use variants of (3)+(4); there is **no peer-reviewed paper specifically on HTML→Figma-component matching**, and the closest academic work is in UI element detection (Screen2Words, Pix2Struct, Apple's Ferret-UI, CMU's WebUI dataset).

## Copilots have converged on a three-layer context model that Mimic should copy

Across Cursor, GitHub Copilot, Windsurf, Claude Code, OpenAI Codex, Replit, Devin, v0, Bolt, and Lovable, the context-and-memory architecture has converged on a **two-or-three-layer file-based model**: **global/user rules** (Cursor User Rules, `~/.claude/CLAUDE.md`, `~/.codex/AGENTS.md`), **project rules** (`.cursor/rules/*.mdc`, `.github/copilot-instructions.md`, project `CLAUDE.md`, `AGENTS.md`), and, in enterprise tools, a **system/org layer**. The file format is typically Markdown with YAML frontmatter carrying `description`, `globs`, and `alwaysApply` (Cursor's `.mdc` format). Activation modes are: always-applied, glob-auto-attached, agent-requested (LLM decides), and manually invoked via `@ruleName`. Cursor, Copilot, Windsurf, Amp, Jules, Factory, OpenAI Codex, and Aider now all honor **AGENTS.md** — stewarded by the Agentic AI Foundation under the Linux Foundation — making it the emerging cross-vendor default. Closest-wins precedence (OpenAI Codex's `AGENTS.override.md` + closest-directory pattern) is the standard resolver.

**"Memory" in these tools is still almost entirely retrieval-based, not model-weight updates.** Cursor Memories (1.0, June 2025) use a sidecar model that observes chats and proposes memories requiring explicit user approval. Windsurf's Cascade auto-generates memories to `~/.codeium/windsurf/memories/` scoped per workspace but recommends users write durable knowledge to `.windsurf/rules/` or `AGENTS.md` instead. Claude Code (v2.1.59+) writes auto-memory to `~/.claude/projects/<project>/memory/MEMORY.md` with topic subfiles. Devin's Knowledge system stores entries as `{description → auto-generated trigger sentence, prompt}` pairs and suggests new additions; retrieval picks based on the description. v0, Bolt, and Lovable lean on project-level context (v0's shadcn registry + Figma import; Lovable's Knowledge File for branding/rules) without persistent auto-memory. **The important design lesson: rules files live in git, so "why did the AI do X?" is answered by `git blame` on a `.mdc` — provenance for free.**

Design tools lag dramatically behind code tools. Figma Make, v0, Galileo AI, Uizard, and Magic Patterns rely on per-session prompts plus library/registry imports; none ships a rules file comparable to `.cursorrules`, and Figma's new MCP rules-file generation is the first parallel. **This is Mimic's second moat**: bring Cursor-grade rules + Graphiti-grade memory into the Figma context.

## Memory architecture: skip Mem0's junk problem, borrow Graphiti's temporal model

The literature and frameworks offer a clear hierarchy of options. **Mem0** (arXiv 2504.19413) runs a two-phase extraction → update pipeline with the LLM choosing ADD/UPDATE/DELETE/NOOP for each candidate fact, backed by a vector DB (Pinecone/Qdrant/pgvector) or optional Neo4j graph. Its new 2026 algorithm switched to ADD-only with multi-signal retrieval (semantic + BM25 + entity). **Critical caveat**: a public GitHub issue (#4573) documents 97.8% "junk" extraction in a 32-day production audit — feedback loops where recalled memories get re-extracted, hallucinated demographics, system-prompt restating. This is the core risk of LLM-driven memory and the reason Mimic should prefer structured, source-tagged writes over open-ended extraction.

**Zep/Graphiti** (arXiv 2501.13956) is the strongest architectural model for Mimic's use case. It maintains a hierarchical, bi-temporal knowledge graph on Neo4j: every edge carries `(t_valid, t_invalid, t_ingested)`, so conflicts **invalidate rather than delete**, enabling full audit and rollback. Retrieval fuses vector + BM25 + graph traversal + centrality rerank with **no LLM calls at retrieval time** (P95 ~300ms) and beats MemGPT on DMR (94.8% vs 93.4%) with 90% less latency on LongMemEval. **Letta/MemGPT** (arXiv 2310.08560) contributes the virtual-memory analogy — core memory blocks pinned in-context + archival vector storage + recall SQL — with **self-editing tools** (`memory_insert`, `memory_replace`, `archival_memory_insert`) the LLM calls to manage its own state. **LangGraph** contributes the namespace model: `(namespace_tuple, key, json_value)` with hierarchical scopes like `("user", "u_1", "project", "p_1", "patterns")`. **A-MEM** (arXiv 2502.12110, NeurIPS 2025) contributes Zettelkasten-style notes with auto-discovered links and memory evolution — adding a new note rewrites contextual descriptions of related old ones. **Generative Agents** (Park et al. 2023) contributes the canonical retrieval score: `α·recency(exp-decay) + β·importance(LLM 1–10) + γ·relevance(cosine)`.

**ChatGPT's Memory**, based on publicly available reverse-engineering (shloked.com, llmrefs.com, Simon Willison), is notably *not* a RAG system — it injects a compressed 5-paragraph "User Knowledge" block into every prompt, regenerated periodically by an LLM summarizer. The bet: for bounded memories, just-inject-all beats retrieval complexity. **For Mimic, this is important**: a user's DS has hundreds of components, not millions, so the "compiled rulebook in every prompt" pattern may beat vector retrieval for the common case and dramatically simplify transparency.

The following table compares the memory frameworks on the axes that matter for Mimic:

| Framework | Storage | Conflict handling | Retrieval | Best fit for Mimic |
|---|---|---|---|---|
| **Mem0 (classic)** | Vector DB + optional KG | LLM ADD/UPDATE/DELETE/NOOP | Top-K semantic | ❌ Junk-extraction risk |
| **Zep/Graphiti** | Neo4j bi-temporal KG | Edge invalidation (non-destructive) | Vector + BM25 + graph | ✅ Audit + rollback |
| **Letta/MemGPT** | Core blocks + archival vector | Self-editing tool calls | Function-call paging | ⚠️ Overkill for rules |
| **LangGraph store** | Namespaced JSON + optional vector | App-level | `put/get/search` | ✅ Simple, inspectable |
| **ChatGPT Memory** | Compressed text block | LLM summarizer regenerates | None — injected | ✅ For small rulebooks |

## Transparency UX: ChatGPT Memory UI, Claude tool-calls, and Cursor rules are the three blueprints

Research across ChatGPT, Claude, Cursor, Copilot, Notion, Linear, Granola, Rewind, and Figma AI yields a pattern catalog where **three implementations stand out**: ChatGPT's enumerated memory list with the inline "Memory updated" pill; Claude's visible `conversation_search` / `recent_chats` tool calls that make retrieval legible in the chat stream; and Cursor's file-as-memory pattern where every rule lives in `.cursor/rules/*.mdc` and `git blame` is the provenance UI.

ChatGPT's Settings → Personalization → Manage memories gives each memory a row with a `⋯` menu (Delete, View history with Restore), a search field, a sort toggle, and a "Delete all" option. New memories trigger a subtle "Memory updated" badge above the response — added explicitly after users complained about silent ingestion. A capacity indicator shows fill percentage against the ~2000-token ceiling; "top of mind" memories can be prioritized; version history enables restoring prior revisions. Claude took a different path: a 24-hour-refreshed **synthesized memory summary** instead of a list, retrieval as explicit tool calls visible in the chat stream, and Pause/Reset as the binary control. Notion AI cites source pages inline on every Q&A response. Microsoft's HAX Guidelines (G8 "efficient dismissal," G9 "efficient correction" with sub-patterns G9-B rich edits and G9-C undo automated actions, G11 "make clear why," G13 "learn from user behavior") and Google PAIR's explainability + feedback chapters give the authoritative framework: **always let users see, edit, and opt out; progressive disclosure; inline provenance; confidence only when tested**.

Synthesized into a set of patterns Mimic should adopt: **enumerated memory inspector** for discrete rules; **inline "pattern learned" notification** after each correction; **visible tool-call provenance** (`match_pattern`, `apply_variable`) so the user sees Mimic reaching into memory; **file-as-memory** with `ds-knowledge.json` in git so diff/blame is free; **two-tier toggle** separating saved pattern rules from synthesized DS insights; **scope hierarchy** (global → org → user → project); **citation-backed recommendations** ("Matched to `Button/Primary` because of rule #42 saved when you corrected `<button class='cta'>` on Oct 14"); and **conversational correction** ("forget the rule about cards" triggers a soft-delete).

## Recommendations for Mimic AI

Mimic's architecture already embodies the right thesis; the gap is operationalizing it with modern memory and UX primitives. The recommendations below are concrete and ordered by leverage.

**Component matching — evolve the catalog pipeline.** Today Mimic's `ds-knowledge.json` appears to record HTML-pattern-to-component mappings discovered at runtime. Upgrade the catalog build to a four-source fusion: (1) REST enumeration of published components and component sets with their `componentPropertyDefinitions`; (2) **Code Connect mapping files** as the highest-authority source for prop shapes and real source paths (this alone will dramatically improve match quality for any team using Code Connect); (3) rendered PNG thumbnails via `exportAsync` for a multimodal index; (4) natural-language docs from Figma component `description` fields. Index with both text embeddings (name + props + description) and CLIP-style image embeddings. At match time, run hybrid retrieval — semantic + BM25 on class tokens + structural feature match on DOM subtree shape — retrieve top-5 candidates, let the LLM pick and emit `importComponentByKeyAsync(key) + setProperties({...})`. This is the same pipeline Graphiti uses and dramatically out-performs pure LLM-in-context or pure heuristic matching.

**Memory layer — adopt a Graphiti-inspired non-destructive schema.** Replace a flat JSON cache with a structured store keyed on pattern signatures and capable of temporal invalidation. Each record:
```json
{ "id": "uuid", "pattern_signature": "button.cta[role=submit]",
  "pattern_embedding": [...], "component_key": "ckey-abc",
  "component_name": "Button", "props_mapping": {"variant":"Primary"},
  "source": "user_correction", "correction_count": 3, "confidence": 0.92,
  "created_at": "...", "valid_until": null, "supersedes": ["id1"],
  "scope": {"project_id":"...","user_id":"...","global":false},
  "examples": [{"html":"...","result_node_id":"..."}] }
```
When a user corrects a mapping, set `valid_until` on the old record, create a new one with `supersedes: [old_id]`, and keep both — this gives audit, rollback, and the ability to answer "why did you choose this last time?" Retrieve with scope cascade (project → user → global), filter to currently-valid records, rerank by `confidence × recency × similarity`. For rule counts under ~500 per project, skip the vector DB entirely and inject a compiled rulebook into the prompt (the ChatGPT pattern) — it is simpler, faster, and fully inspectable.

**Learning signals — treat corrections as first-class, avoid open-ended extraction.** The Mem0 junk lesson is decisive: do **not** run an LLM extractor that scans every interaction for "things worth remembering." Instead, make learning event-driven with three high-signal triggers: (1) **explicit user correction** ("this should have been a `Card`, not a `Frame`") writes a new record with confidence 0.9+; (2) **explicit user confirmation** ("keep using `Button/Primary` for `.cta`") promotes an existing pattern from CANDIDATE to VERIFIED; (3) **pattern repetition** — if Mimic makes the same match three runs without correction, auto-promote. Add a fourth state — NOT_WORTH_STORING — so the LLM can opt out. Keep the Mimic reporting pattern of "patterns saved / promoted to VERIFIED / reads used" as the core learning summary.

**Transparency UX — ship a memory inspector and inline notifications.** Mimic's Claude Code surface is a terminal + a Figma plugin, so the memory inspector should live in the plugin UI. The minimum viable set:
- A **Rules panel** in the plugin showing all learned patterns as rows — pattern signature, mapped component, source (user correction / auto-inferred / verified), confidence, last used, `⋯` menu with Delete / View history / Supersede.
- A **"Pattern learned"** inline toast after each Mimic run ("Saved: `<div class='card-*'>` → `Card`. [View rules]").
- **Provenance in outputs** — after each frame is built, report "Used rule #42 (`button.cta` → `Button/Primary`), saved Oct 14 from your correction."
- **`git`-first storage** — keep `ds-knowledge.json` human-readable and commit-friendly; optionally shard into `.mimic/patterns/*.json` per the Cursor `.mdc` pattern so diffs stay tractable. This gives every team `git blame` for provenance.
- **Scope controls** — three tabs: Project / User / Global. Mirror Cursor's User-Rules-vs-Project-Rules UX and honor AGENTS.md / CLAUDE.md so Mimic participates in the wider copilot ecosystem.
- **Conversational correction** via Claude Code: "forget the rule about cards" should trigger a supersede, not a silent delete.
- **A DS-gap report** — Mimic already detects when HTML patterns have no matching component. Surface this as a standing "DS Gaps" list so design leads can use Mimic as an audit tool, not just a converter.

**Market gaps Mimic should fill.** The research reveals three unoccupied positions. First, **code-to-Figma with DS preservation is a blank space** — every AI-era entrant points the other direction. Second, **"learning design copilot" is an unclaimed label**: Figma Make, v0, Galileo, and Magic Patterns all lack rules files, lack persistent memory, and lack provenance. Third, **DS auditing as a byproduct**: the gap-detection Mimic already surfaces is something Supernova and zeroheight sell as standalone products. Positioning Mimic as "the MCP that learns your design system while it builds, and tells you where it's broken" captures all three.

**Competitive counters Mimic must match.** Builder.io Visual Copilot's catalog-based matching is its strongest feature and Mimic should match it via the Code Connect fusion above. Figma's official MCP will commoditize basic read/write; Mimic must make its compounding-learning lead visible through the UI patterns above — if users can't see Mimic learn, they won't believe it is. Anthropic's Claude Design (April 2026) competes upstream of Mimic by generating prototypes conversationally, but critically does **not** export to Figma; Mimic should position as the step that turns Claude-generated HTML into DS-compliant Figma files, and integrate tightly with Claude Code so the handoff is seamless.

## Conclusion

Mimic AI's bet — that HTML-to-Figma with compounding DS-aware learning is an underserved, defensible niche — is correct and supported by the research. The tool already solves the Figma read-quota problem better than anything else in the market. The remaining work is not architectural novelty but **operational maturity on two axes**: a memory layer that non-destructively records, supersedes, and audits every learned pattern (Graphiti's bi-temporal model plus LangGraph's namespace hierarchy plus Code Connect's component catalog), and a transparency UX that makes those learnings visible, editable, and explainable (ChatGPT's enumerated inspector plus Claude's visible tool calls plus Cursor's file-as-memory). These patterns are now standard in code copilots and almost entirely absent in design copilots; adopting them would move Mimic from "clever MCP script" to "the design-system memory layer for AI-native teams" — a category name no competitor currently owns.
