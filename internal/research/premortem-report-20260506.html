Mimic AI v2 — Pre-Mortem Report

7 failure scenarios analyzed in parallel / 2026-05-06 / Design spec stress test

Synthesis

Most Likely Failure

LLM orchestration collapse on complex builds. 45 tools, a 7-step component configuration process, native chart math, and 6 sequential phases — guided by only 8 rules and contextual hints. Simple builds (a card, a form) will work. Dashboards with 10+ sections, multiple chart types, and 30+ components will break in unpredictable ways: skipped phases, half-configured components, wrong chart proportions. The tool will produce "something" that's always incomplete in ways users can't anticipate. This is the most likely failure because it happens on every complex build, not just edge cases.

Most Dangerous Failure

Plugin enforcement gate kills adoption on imperfect design systems. If the plugin rejects every text node without a text style and every fill without a color variable, and 80% of real-world DSs have gaps in their token coverage, then 80% of first-time users get a wall of errors and zero output. They don't come back. They tell others "it doesn't work." This is the most dangerous because it's invisible (users who bounce don't file issues), irreversible (first impressions are permanent), and attacks the core value prop (DS compliance becomes DS gatekeeping).

The Hidden Assumption

The spec assumes the user's DS is structurally complete. Every enforcement decision, every error message, every compliance check assumes there's a DS text style for every text size and a color variable for every fill. But real design systems are works in progress. Some have 200 components and 3 color variables. Some have 500 variables and 0 text styles. Some have everything but use unconventional naming. The entire enforcement hierarchy was designed for the ideal DS, not the real one.

Revised Plan

Five specific changes to the spec:

1. Add a chart calculation layer in the MCP server. Don't ask the LLM to compute radian angles, SVG bezier paths, or trig. The MCP server receives data + chart type and returns the exact coordinates, SVG strings, and arc angles. The LLM just passes them to the plugin. Math in Node.js, not in the prompt.

2. Make enforcement graduated, not binary. Phase 2 inventories what the DS actually has. The plugin enforcement gate adapts: if the DS has text styles, require them. If it doesn't, accept raw text and flag it. Same for colors, spacing, radius. The enforcement level is set per-category based on what the DS provides, not a global strict/permissive switch.

3. Add phase enforcement to the MCP layer. Don't rely on tool hints alone. The MCP server tracks the current phase. If Phase 1 hasn't completed, Phase 3 tools return an error: "DS Discovery not complete. Call mimic_discover_ds first." Mechanical enforcement of sequencing, not advisory hints.

4. Add WebSocket keepalive + auto-reconnect. Ping every 15 seconds. If the connection drops, auto-reconnect with a 3-attempt backoff. The plugin maintains a pending-operation queue so in-flight calls resume after reconnect. Surface connection status in mimic_status.

5. Make the first build succeed on any DS. Even if the DS has zero tokens, the first build should produce a complete artboard using whatever the DS provides (components at minimum) and raw values where tokens don't exist. The build report then says: "Built with 12 components and 0 DS tokens. Adding color variables would enable full DS compliance." The tool demonstrates value first, then recommends improvements. Never block the first build.

Pre-Build Checklist
  • Chart math validation: Before shipping, test every chart type (bar, donut, line, radar, scatter) with known data and verify pixel-level correctness. If the MCP calculation layer produces wrong output, it's a code bug you can fix — not an LLM hallucination you can't.
  • Imperfect DS test suite: Test with 4 real-world DS configurations: (a) full tokens, (b) components only, (c) partial tokens (colors yes, spacing no), (d) community library with non-standard naming. All 4 must produce usable output on first build.
  • Complex build stress test: Build a 12-section dashboard with 3 chart types and 20+ component instances. Verify all 6 phases complete, all components are configured, no placeholder text survives. Run this 3 times — if it fails once, the orchestration isn't reliable.
  • WebSocket durability test: Run a build that takes 5+ minutes (200+ tool calls). Verify the connection doesn't drop. Simulate a drop mid-build and verify auto-reconnect works.
  • First-run onboarding test: Have a non-technical designer (not the developer) attempt the full setup from zero. Time each step. If any step takes >2 minutes or requires Googling, it needs documentation or simplification.

7 investigators analyzed failure scenarios independently

Plugin Enforcement Too Rigid
Critical
Failure Story

First users aren't designers with manicured DSs. They have community libraries with 200 color variables but no semantic aliases, text styles that cover headings but not body copy. The plugin fires rejections on every body text node, every background fill. Zero output. A wall of errors with no path forward. Word spreads: "Mimic doesn't work." The enforcement gate — the core differentiator — becomes the core complaint.

Underlying Assumption

The plugin's binary enforcement assumes target design systems are structurally complete enough to cover every token category encountered in real HTML.

Early Warning Signs

Beta users report zero-output builds on first attempt with 10+ errors. Support requests cluster around "my DS has X but not Y."

Component Matching Unreliable
High
Failure Story

DS has the button named "CTA/Primary" nested under "Interactions/Actions." Search for "button" returns "Button Group" or nothing. Mimic picks the wrong match and proceeds. Worse: the cache compounds the error — every subsequent build confidently uses the wrong mapping. Users from agencies with inherited client DSs have significantly lower satisfaction than users building on their own DS.

Underlying Assumption

Component names in real design systems are predictable enough that keyword search can reliably identify the canonical component.

Early Warning Signs

High cache confidence scores alongside high user correction rates on the same builds — the tool is certain and wrong simultaneously.

LLM Orchestration Collapse
Critical
Failure Story

User asks for a 12-section dashboard. LLM completes DS Discovery and jumps to Build, skipping Style Inventory. The 7-step component process falls apart: after inserting a component, the LLM runs step 1 (get props), maybe step 2 (set variant), then drifts to the next insertion without finishing configuration. Charts are built without labels. "It builds something, but it's always incomplete in unpredictable ways." Worse than a hard failure — unreliable in ways users can't anticipate.

Underlying Assumption

Tool response hints can substitute for explicit state tracking, but hints are advisory — working memory pressure causes the LLM to act on recency rather than protocol.

Early Warning Signs

Phase skip rate >15% on builds with 6+ sections. Component configuration completeness drops below 70% on dashboards with 3+ chart types.

WebSocket Bridge Silent Failure
High
Failure Story

Complex dashboard: 8-10 minutes of tool calls succeed, then the WebSocket drops silently at the 2-3 minute mark. The MCP server thinks it's connected. The plugin has no reconnect. The user sees cryptic timeouts. Because the bridge is embedded, there's no separate process to restart. Users must kill everything and start over. Partial builds leave orphaned frames. "Good for small screens, unreliable for real work."

Underlying Assumption

A WebSocket initiated at build start will remain stable for any realistic build duration without keepalive or reconnect logic.

Early Warning Signs

Build success rate drops sharply for jobs >150 tool calls. Users report "restart the plugin" on complex builds but succeed on simple ones.

First-Time Experience Too Complex
High
Failure Story

Setup requires: terminal, npx, JSON config editing, plugin install from manifest, library enable. Each step is a filter. Designers who hit a terminal error at step 1 have no recovery path. A stray comma in JSON causes silent failure. The 607 weekly downloads were self-selected developers. The audience shift to designers was never matched by a delivery shift. Reviews: "incredible once it works" and "I gave up." The second group is larger and quieter.

Underlying Assumption

Designers who feel the pain of DS inconsistency are willing and able to complete a developer-grade setup process if the promise is compelling enough.

Early Warning Signs

Support requests cluster around JSON config step, not tool behavior. Install completion rate differs sharply by OS and technical background.

Learning Loop Never Reaches Critical Mass
Medium
Failure Story

Designers run 2 builds in the first week, then return to normal workflow. Three weeks later the DS has changed, fingerprints don't match, recipes are stale. The experience isn't faster — it's a re-explanation of why the knowledge is stale. The auto-promote threshold of 3 builds sounds low until most users never hit 3 total. Gap recommendations are never actioned — users read the report and close it. The learning loop is infrastructure that never exercises.

Underlying Assumption

Designers return to the same tool repeatedly within a timeframe short enough that accumulated knowledge remains valid.

Early Warning Signs

Median builds-per-user-per-month stays at 1-2. Gap recommendations are never actioned or forwarded to DS owners.

Native Chart Building Math Failures
Critical
Failure Story

Bars where 42% renders taller than 78%. Donut segments summing to 340° or 390°. Radar polygons lopsided at the 4th vertex. SVG paths with coordinates off-canvas. Every broken chart costs ~25 tool calls — 25x the old convenience tool. Users retry, multiplying the burn. "100% DS compliant" becomes a bitter joke when charts are geometrically incoherent. The root cause can't be fixed because LLMs can't be made reliably better at floating-point geometry through prompting.

Underlying Assumption

LLMs can reliably perform multi-step floating-point geometry (angle accumulation, trig, SVG path construction) at production quality.

Early Warning Signs

Donut segment angles need manual correction in >30% of runs. Token consumption per chart averages 3x estimates due to retries.
