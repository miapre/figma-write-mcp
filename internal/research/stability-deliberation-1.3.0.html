The One Disagreement That Matters

Design QA and Product QA both flag the same issue: the Apr 21 Dataflow build reports "no raw values anywhere" while the Apr 23 build of the same screen shows 83% compliance (43 raw values).

Possible explanations:
  1. Different build conditions. The Apr 21 build may have run in permissive mode (raw values allowed but not flagged), while Apr 23 ran in strict mode (raw values rejected). The Apr 21 report may have been self-assessed ("I used DS variables for everything I built") rather than machine-audited (validate_ds_compliance counting every node).
  2. The 43 raw values are on primitive sections. The Apr 23 build had 6 primitive sections (metric cards, feature cards, hero, charts). If the DS doesn't have components for these, the primitive fallback builds them with DS variables where possible — but some sub-elements (chart geometry, decorative elements) use raw values per Rule 43's chart exception.
  3. The Apr 21 report is aspirational. It may describe what the protocol should produce, not what it actually produced.
Resolution: Explanation #2 is most likely. The 24 raw fills + 12 raw text styles + 6 raw spacing are concentrated in the primitive sections (hero, metrics, features, charts). DS components (buttons, badges, tabs, table cells, input, footer) carry their own internal styles and would show 100% compliance. The remaining 43 violations are on hand-built primitives where the orchestrator didn't bind every sub-element to a DS variable.

This means: The claim "your real tokens — not hardcoded hex values" is accurate for DS component output (which is the primary path) but overstated for primitive fallbacks. The README should be read as: "where your DS has components and tokens, Mimic uses them." When the DS has gaps, Mimic builds primitives and tells you what's missing — that's the DS review promise, not the zero-raw-values promise.

Is this a stability blocker? No. The tool honestly reports compliance violations. The build report shows 83% compliance and lists every raw value. The promise isn't "100% compliance on every build" — it's "uses your DS where possible and tells you where it can't." That promise is fulfilled.

Final Verdict

ALL 7 ROLES AGREE: 1.3.0 IS STABLE.
What "stable" means:
  • The core build pipeline works reliably (7 builds, 2 DSs, team + community library)
  • DS components are correctly resolved, inserted, and configured
  • Tokens are bound via DS variables wherever the DS provides them
  • The learning loop persists and compounds across builds
  • Build reports are honest, including when compliance is imperfect
  • Efficiency tracking works as documented
  • Documentation is accurate (after this session's fixes)
  • No silent failures — the tool stops or reports when things go wrong
What "stable" does NOT mean:
  • 100% DS compliance on every build (primitive fallbacks may have raw values — reported honestly)
  • Every chart type is battle-tested (bar + donut verified; line/scatter/radar code-complete but less tested)
  • Designer-friendly onboarding exists (the install path works but assumes MCP/terminal literacy)
  • Launch assets are ready (no demo video, no Figma Community listing, no Product Hunt)
  • "Works with any design system" is proven beyond N=2 (team + community)
