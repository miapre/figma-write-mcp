1B — Warm the Cache

Critical for a flawless build. Run these commands in Claude Code BEFORE recording. This ensures fonts are loaded, styles are cached, and component keys are resolved. A cold-start build will pause on font loading and style discovery — looks bad on camera. 
# In Claude Code, before recording:

1. "Run mimic_status" → Confirm bridge: connected, plugin: connected
2. "Run mimic_discover_ds for the Untitled UI library"
   → This caches all component keys, text styles, and variable names
3. "Preload all text styles from UUI"
   → figma_preload_styles with the UUI style keys
4. "Set session defaults: dsMode strict, textFillStyleKey for text-primary"
   → figma_set_session_defaults

After these 4 commands, the cache is warm. The actual build will use cached patterns and skip discovery.

Part 5 — Failure Recovery Plan

If something goes wrong during recording:

FailureRecovery
Bridge disconnectsStop recording. Restart bridge (npm run bridge). Re-run plugin. Resume from the section that failed.
Component import times outStop recording. The component key may be stale. Run mimic_discover_ds to refresh. Retry.
Strict mode violation during buildThis IS the tool working correctly — it stops you from using raw values. If it happens, the orchestrator should find the right DS variable and retry. Don't hide it; it's a feature.
Table cell text shows "Label"Phase 4 catches this. The build should fix it before reporting "done." If it slips through, note it and fix in post — or re-record that section.
Charts look wrongBar chart is reliable. Donut chart was just fixed (1.3.0). If donut still fails, skip it and crop the GIF to exclude it. A partial demo is better than a broken one.
Build takes too longEdit speed. Real-time build of this page is ~3-5 minutes. At 4x speed that's 45-75 seconds of GIF — crop the middle sections and show prompt → table → report.

Part 6 — Dry Run Validation

Before recording, do a full dry-run build. Same HTML, same DS, same page. This catches any issue the recording would hit. Delete the dry-run artboard after validation. The recording build will be the clean one.

Dry-run checklist (all must pass):

  • Header nav component inserts and configures correctly (logo, links, buttons)
  • Badge shows "New — Real-time pipelines in beta" (not default text)
  • Buttons show correct labels (not "Button CTA")
  • Tabs show "Ingestion / Pipelines / Visualization / Alerts" (not default labels)
  • Table header cells show "Feature / Free / Pro / Enterprise"
  • Table cells show actual data (not "Label" or "Olivia Rhye")
  • Input shows placeholder "your@company.com"
  • Footer shows "Dataflow" brand + 3 link columns
  • Bar chart has 12 bars with DS color fills
  • Donut chart renders (3 segments, legend with percentages)
  • validate_ds_compliance returns acceptable compliance (100% on DS components, some violations on primitives is OK)
  • Build report generates with Efficiency section
  • Phase 5 summary shows in terminal
  • Total build time is under 5 minutes

If the dry run passes: Delete the artboard, start the screen recorder, and do the real build.

If the dry run fails: Fix the issue, dry-run again. Do not record until a clean run completes.
