Pillar 1 — Builds with your design system

Core build

FeatureWhat it does
HTML → FigmaTakes any HTML (file, URL, or raw markup) and produces a structured Figma artboard using your DS components, tokens, and text styles.
Vibe design (text → Figma)Describe a screen in plain text ("Build a dashboard with metrics and a data table") — Mimic generates a DS-compliant artboard without any HTML input.
URL renderingGive it a URL. Mimic renders the page (including client-side JS via Puppeteer), extracts the HTML, and builds from it. Handles SPAs and auth-gated pages.
Multi-page HTMLWhen the HTML contains multiple views/pages, Mimic lists them and lets you choose which to build. Builds one at a time, learns between each.

DS component usage

FeatureWhat it does
Mandatory DS discoveryBefore every build, searches the DS for every component type in the HTML (buttons, tabs, badges, table cells, pagination, inputs, page headers, nav, cards, etc.). Produces a component map.
Real component insertionImports actual DS components by key — not drawn rectangles. Configures variant, size, state, text overrides, icon visibility, semantic colors.
Component configurationFull post-insertion setup: correct variant selected, all text overridden (no "Label" placeholders), icon slots hidden if unused, semantic color properties set, boolean feature toggles configured.
Component set verificationWhen multiple sets have identical variant names (Button vs Button destructive vs Button success), verifies the component SET name, not just the variant.
Safe primitive fallbackWhen the DS has no matching component, builds from primitives using only DS variables. Documents the reason and the search terms used.

DS token binding

FeatureWhat it does
Color variable bindingEvery fill and stroke bound to DS color variables via setBoundVariableForPaint. Semantic category enforcement: text colors for text, background colors for frames, border colors for strokes, foreground colors for icons.
Text style bindingEvery text node gets a textStyleId from the DS. No raw fontName, fontSize, fontWeight, lineHeight. The text style is the single source of truth.
Spacing variable bindingEvery padding, gap, and margin bound to DS spacing variables via setBoundVariable. Bound at creation time, not as a post-build fix pass.
Radius variable bindingEvery cornerRadius bound to DS radius variables. Bound at creation time.
Strict modedsMode: "strict" — all tools reject raw hex, raw px, and raw font values. Throws DS_STRICT_VIOLATION. Forces DS-only output.
Permissive modedsMode: "permissive" — allows raw fallbacks for component-only DSs without published tokens. Documents gaps in report.

Auto-layout

FeatureWhat it does
Auto-layout on every frameEvery content frame uses auto-layout (VERTICAL or HORIZONTAL). Hug or fill — no fixed heights on content. Only the artboard is 1440px FIXED width.
CSS → Figma layout translationReads the HTML's CSS and maps directly: flex-direction → direction, gap → itemSpacing, padding → padding with DS variables, max-width → Figma maxWidth + parent CENTER, flex:1 → FILL.
Container fidelityExtracts the HTML's container model (max-width, padding, gap, centering) and applies it using DS variables. No side-padding hacks.

Charts

FeatureWhat it does
Bar chartsAuto-layout horizontal frame, bars use layoutGrow:1 for even distribution, bottom-aligned, DS color fills.
Donut/pie chartsArc geometry with DS color variables per segment. Legend with auto-layout. categoryVariables maps DS colors to data entries.
Line / scatter / radarSupported in code. SVG-based for complex geometry. DS colors for all elements.
Chart containersAuto-layout wrappers (cards, headers, legends) — same rules as any other frame. Only chart geometry (arcs, dots, paths) may use absolute positioning.

Multi-DS support

FeatureWhat it does
Team/org librariesFull support: components, text styles, color/spacing/radius variables. Full DS compliance.
Community librariesFull support with key-based import. Variables discovered via REST API, imported via importVariableByKeyAsync. Components and text styles import normally.
Component-only DSsSupported. Components used where available. Raw values for text/colors/spacing (no tokens exist). Report recommends adding token collections.
Restyle artboardDuplicate an artboard and restyle it with a different color map, font, and radius scale — showing the same layout built for a different DS.

Pillar 2 — Reports and learns from every build

Build reports

FeatureWhat it does
Structured build reportAfter every build: what was built, which DS components were used, which elements were primitives and why, quality metrics (raw values count), issue log.
DS gap recommendationsReports which components the DS is missing, with evidence: element count, build count, search terms used. Phrased as questions: "Should your DS include a Metric Card?"
Efficiency sectionTool call count, cache hits, saved-vs-cold estimate, DS component call savings, per-gap savings projection ("Adding a Metric Card would save ~20 calls/build").
Compliance auditvalidate_ds_compliance post-build: counts raw fills, raw text styles, raw spacing, fixed sizing. Any violation is a build defect.
HTML or Markdown exportReports saved to mimic/reports/ as styled HTML or markdown. Shareable with the team.
Cross-build comparisonIf a similar screen was built before, notes improvements and regressions.

Learning loop

FeatureWhat it does
Pattern persistenceComponent mappings saved to ds-knowledge.json with confidence levels. Accumulates across builds.
User correctionTell Mimic "that's not the right Badge — use Tag/Neutral" and the mapping updates permanently. Confidence 0.9, supersedes old pattern.
Auto-promotionIf the same mapping goes uncorrected for 3 builds, it auto-promotes from CANDIDATE to VERIFIED.
Configuration recipesRemembers HOW to configure a component — which variant, which text overrides, which icon slots to hide. Replays on next build, skipping inspection.
Warm-cache buildsCached patterns validated against live DS before use. Valid matches skip DS search entirely. Invalid matches re-searched.
Pattern-learned notificationsAfter every build: "Learned: button/secondary → Button (md, Secondary gray). Promoted: table/cell-badge (3 builds). Cache: 20 verified."
DS change detectionAt build start, compares DS against last known state. New components surfaced. Removed components warned. Stale cache invalidated.
Regression checkIf a previous build used a DS component for an element, this build must too. Using a primitive where a component was used before is flagged.
Cost reduction curveCold build ~140 calls → warm ~80 → hot ~55. Each build costs less as patterns cache and decisions become instant.

Pillar 3 — Prepares your DS for AI (without extra effort)

FeatureWhat it does
Component description suggestionsTracks how components are actually used across builds — which variants, in which contexts, for what purpose. Generates descriptions: "Button: Primary action trigger. Used for CTAs (Contained/Primary), secondary actions (Outlined). Sizes: sm for nav, md for forms, lg for hero." Review and add to your library.
DESIGN.md generationCompiles your DS into the open DESIGN.md format: color tokens, typography scale, spacing, radius, component patterns. Machine-readable YAML frontmatter + human-readable usage guidelines. Compatible with Stitch, Cursor, Copilot, generative UI frameworks.
DS gap tracking (cumulative)Gaps maintained across builds in ds-knowledge.json. After 10 builds, Mimic knows exactly which components your DS needs — with usage counts, build counts, and savings estimates.
Token gap detectionWhen a DS has components but no published tokens, reports it: "Your library has no published design tokens. Adding color, spacing, and radius variable collections would enable full token binding and mode support (light/dark)."

Infrastructure

Architecture

FeatureWhat it does
MCP server45 tools exposed via Model Context Protocol. Works with Claude Code, Cursor, VS Code, Windsurf, JetBrains.
Local bridgeHTTP/WebSocket bridge between MCP server and Figma plugin. Runs on localhost:3055.
Figma pluginPlugin sandbox executes instructions from the bridge. Handles component import, style application, variable binding.
DS-agnosticZero hardcoded DS values in code. All DS-specific data comes from runtime discovery. Works with any published Figma library.

Quality governance

FeatureWhat it does
46 golden rulesEnforced on every build. DS-first, auto-layout, content fidelity, no UI invention, zero raw values, mandatory discovery, build reports, mandatory stop on failure.
6-phase build lifecyclePhase 0 (Target) → Phase 1 (DS Discovery) → Phase 2 (Style Inventory) → Phase 3 (Build) → Phase 4 (QA) → Phase 5 (Report). Each phase is a gate.
7 rolesPlatform Architect, Build Engineer, Design QA, DS Integration Engineer, Learning Engineer, Product QA, Marketing. Each owns specific phases and rules.
Mandatory stop protocolBuild stops (not silently degrades) when: DS library unreachable, zero components resolved, same error 3 times. Partial report saved.

Privacy and trust

FeatureWhat it does
Runs locallyEverything on your machine. Bridge, plugin, MCP server — all localhost.
No telemetryZero analytics, zero tracking, zero outbound calls except Figma REST API for component keys.
Open sourceMIT license. Full source on GitHub.
Design data stays localComponent names, token values, HTML content — never sent anywhere except Figma's own API.
